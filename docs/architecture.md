# Architecture

OpenGriffin is a single Python process with three concentric loops:

1. **The chat loop** — a long-running asyncio process that talks to one or more messaging gateways (Telegram by default).
2. **The cron + trigger loop** — APScheduler firing scheduled jobs and ambient triggers in the same event loop.
3. **The compounding loop** — nightly auto-jobs (journal, dream, drift, voice card refresh) that improve the agent's persistent state without prompting.

All three share one address space, one set of MCP servers, one filesystem state directory.

## Process model

```
opengriffin run
├── asyncio event loop
│   ├── Telegram polling (or webhook)
│   ├── Discord/Slack/Email/iMessage/Signal/Matrix gateway adapters (opt-in)
│   ├── Webhook HTTP server (port 8645)
│   ├── Observability HTTP server (port 8646)
│   ├── A2A HTTP routes (mounted on 8645)
│   ├── APScheduler with all cron/trigger/timelock jobs
│   └── Worker pool tasks (one per spawned worker)
└── Claude Agent SDK subprocess
    ├── Loads ~/.claude/skills/ + ~/.claude/credentials.json
    ├── Hosts in-process MCP servers (memory, kanban, recall, …)
    └── Stores sessions at ~/.claude/projects/<encoded-cwd>/<session_id>.jsonl
```

## Module overview

The agent is intentionally split into ~30 single-purpose modules under `src/opengriffin/`. Each module:

- Owns one feature
- Has its own state file (or appends to a shared log)
- Exports an MCP server registered into `bot.build_mcp_servers()`
- Is independently testable

See [`CONTRIBUTING.md`](../CONTRIBUTING.md#module-shape) for the canonical module shape.

## How a Telegram message flows

```
User → Telegram → bot.on_message
  → deadman.heartbeat() (resets timeout)
  → predictive.lookup_cached() (instant if pre-computed)
  → progress.start() → spawn heartbeat task
  → claude_agent_sdk.ClaudeSDKClient
      ├── system prompt = constraints + memory + sysprompt
      ├── mcp_servers = 30+ in-process MCP servers
      ├── can_use_tool = approvals.can_use_tool (callback)
      └── hooks = checkpoints.HOOKS_SPEC (snapshots before Write/Edit)
  → stream messages → emit observability events
  → on each ToolUseBlock: progress status updated
  → on AssistantMessage: text accumulated
  → on ResultMessage: session_id persisted, cost logged
  → reply edited into the original status message
```

## Why this shape

- **One process, no daemons** — easy to launchd / systemd / Docker.
- **No backend** — every feature works offline (except where it explicitly fetches a provider).
- **Plain files** — every state is greppable, editable, version-controllable.
- **MCP-native** — every internal feature is also exposed as a tool the LLM can call autonomously, not a hidden side effect.
